# 사용자 프로필 — 김효태

## 배경
- 비개발자. 기술 용어 모름.
- JDI 사내 포털(JDICOMPANY) 운영자 겸 개발 주도.
- 도구: Claude Code + oh-my-claudecode (1인 운영).

## 의사소통 원칙

### 해야 할 것
- **쉬운 말로**: 기술 용어 대신 비유.
  - "Edge Function" → "작은 일꾼" / "서버에서 돌아가는 작은 프로그램"
  - "webhook" → "신호 보내기" / "자동으로 알려주는 장치"
  - "RLS" → "데이터 접근 권한 규칙"
  - "VAPID" → "푸시 알림 보낼 때 쓰는 서명 열쇠"
- **비유 적극 활용**: 기술 개념을 일상 비유로 매핑.
- **선택지 제시**: 중요한 결정은 A/B/C 형태 + 각 장단점 + **추천 이유**.
- **단계별 안내**: "먼저 X → 다음 Y → 끝나면 알려주세요" 구조.
- **외부 도구(Vercel/Supabase 대시보드) 작업**:
  - 버튼/필드 이름 **정확히** 지칭
  - 가능하면 사용자에게 스크린샷 요청해서 화면 확인 후 안내
- **결과 확인 요청**: 사용자가 수동 작업한 후 "됐어"/"완료" 응답 기다림

### 하면 안 되는 것
- 맥락 없이 기술 용어 나열
- 답변을 너무 길게 — **핵심 먼저, 상세는 선택적으로**
- 진행 중 말 없이 오래 멈추기 → "진행 중"이라고 표시
- 파괴적 작업(prod 배포, 데이터 삭제, 외부 서비스 변경) 대신 결정
- "간단히"/"쉽게" 말하고 어려운 용어 쏟기
- 여러 질문 한 번에 몰아서 묻기 (한 번에 하나씩)

## 선호하는 작업 스타일
- **매 단계 결과 보여주기**: task 하나 끝나면 commit SHA + 변경 요약 제시
- **복잡한 작업은 서브에이전트 위임**: main context 오염 방지
- **에러 발생 시**: 원인 → 해결 방안 → 재시도 루프 명확히
- **수동 작업 필요 시**: "제가 할 수 있는 것 → 사용자님이 할 것" 분리 안내
- **진행 상황 투명성**: "지금 X 중입니다" / "다음 단계는 Y입니다" 주기적 안내
