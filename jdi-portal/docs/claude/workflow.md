# 작업 흐름

작업이 작고 명확하면 이 문서를 읽지 말고 바로 관련 파일을 확인해 처리합니다. 여러 도메인에 걸치거나 요구사항이 불명확할 때만 아래 흐름을 사용합니다.

## 1. 이해

- 사용자의 실제 목표를 한 문장으로 정리합니다.
- 영향 범위를 도메인 단위로 나눕니다. 예: 근태, 업무, 채팅, 일정, 인플루언서, Supabase.
- 운영 데이터, 권한, 배포에 영향이 있으면 먼저 위험을 확인합니다.

## 2. 설계

큰 기능은 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`에 설계를 남깁니다.

설계 문서에는 다음을 포함합니다.

- 사용자 흐름
- 바뀌는 화면과 데이터
- 필요한 테이블/RPC/정책
- 검증 방법
- 위험과 롤백 방법

## 3. 계획

구현이 여러 단계라면 `docs/superpowers/plans/YYYY-MM-DD-<topic>.md`에 작은 작업 단위로 나눕니다.

좋은 작업 단위:

- 한 번에 이해할 수 있습니다.
- 변경 파일 범위가 명확합니다.
- 실행할 검증 명령이 있습니다.
- 이전 단계 결과에만 의존합니다.

## 4. 구현

- 기존 도메인 패턴을 먼저 따릅니다.
- 서버/클라이언트 경계를 흐리지 않습니다.
- RLS와 권한 검증을 DB와 서버 양쪽에서 확인합니다.
- UI 문구는 쉬운 한국어로 씁니다.
- 큰 리팩터링은 요청 범위를 벗어나면 하지 않습니다.

## 5. 검증

기본 검증:

```bash
npm run lint
npm run build
```

변경 유형별 추가 확인:

| 변경 | 확인 |
|---|---|
| DB/RLS | 마이그레이션 순서, 정책 조건, 관리자/일반 사용자 접근 |
| Edge Function | Deno 호환성, secrets, 배포 명령 |
| Realtime | 구독 cleanup, 중복 이벤트, 권한 |
| 날짜/근태 | KST 경계, 월말/자정, 휴가 연동 |
| 파일 업로드 | 크기, 확장자, Storage 정책 |
| UI | 모바일/데스크톱, 로딩/빈 상태/오류 상태 |

## 6. 보고

최종 보고에는 다음만 간결히 포함합니다.

- 무엇을 바꿨는지
- 어떤 검증을 했는지
- 남은 위험이나 확인하지 못한 점

사용자가 원하지 않으면 장문의 내부 과정은 생략합니다.
