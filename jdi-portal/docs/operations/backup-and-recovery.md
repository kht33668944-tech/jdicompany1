# 백업·복구·장애 대응 운영 가이드

> 대상: 비개발자 운영자. 화면에서 어디를 눌러 무엇을 확인하는지 위주로 적었습니다.
> 원칙: **실제 복구·삭제는 운영 DB에서 바로 하지 말고, 반드시 별도 환경에서 먼저 검증**합니다.
> (이 문서는 코드 저장소의 사실만 담고, 대시보드에서만 알 수 있는 항목은 "대시보드 확인 필요"로 표시합니다.)

---

## 0. 지금 코드로 확정할 수 있는 것 / 없는 것

**코드로 확정되는 사실**
- DB 스키마(테이블 구조·정책)는 `supabase/migrations/`에 번호순으로 **버전 관리**되어 있습니다. 즉 "구조"는 git에 백업돼 있습니다.
- 저장소 안에는 **데이터 자동 백업(pg_dump 등) 스크립트가 없습니다.** 즉 실데이터 백업은 전적으로 Supabase 플랫폼 기능에 의존합니다.
- `supabase/config.toml`은 **로컬 개발용** 설정입니다. 운영 백업 주기·보존 기간은 여기서 알 수 없습니다.

**대시보드에서만 확인 가능(추측 금지)**
- 운영 DB의 자동 백업 켜짐 여부, 보존 기간, 시점 복구(PITR) 가능 여부 → **Supabase 대시보드 확인 필요**.
- Railway 배포 알림·로그 보존 → **Railway 대시보드 확인 필요**.

---

## 1. 백업 상태 확인 방법 (Supabase 대시보드)

1. https://supabase.com 로그인 → 이 프로젝트(JDI 포털) 선택.
2. 왼쪽 메뉴 **Database → Backups** 클릭.
   - **Daily backups**: 매일 자동 백업 목록과 날짜가 보이면 켜져 있는 것입니다. 보존 기간을 확인하세요.
     - 참고(플랜별 일반 정책, 실제 값은 화면 기준): **Free 플랜은 자동 백업이 없을 수 있습니다.** Pro 플랜은 보통 **7일** 보존, 상위 플랜/추가 옵션에서 더 김.
   - **Point in Time Recovery(PITR)**: "특정 시각으로 복구" 기능. 켜져 있으면 분 단위 복구가 가능합니다(유료 옵션일 수 있음).
3. 화면 상단/프로젝트 설정 **Settings → Billing** 에서 현재 플랜을 확인하면 백업 정책을 가늠할 수 있습니다.

> ✅ 오늘 할 일: 위 **Database → Backups** 화면을 열어 (1) 자동 백업이 있는지 (2) 며칠 보존인지 (3) PITR이 켜져 있는지 — 이 3가지를 메모해 두세요. 실제 백업 값·비밀정보는 저장/공유하지 마세요.

## 2. 데이터 복구가 필요할 때의 안전한 절차

> ⚠️ **가장 중요한 원칙**: 운영 DB에 바로 복원 버튼을 누르면, 잘못된 백업을 덮어써 **현재 데이터가 사라질 수 있습니다.** 아래 순서를 지키세요.

1. **멈추기**: 문제를 인지하면 우선 상황을 파악합니다. 절대 급하게 "복원"을 누르지 않습니다.
2. **범위 파악**: 무엇이(어느 테이블/어느 사용자) 언제부터 잘못됐는지 확인합니다. (예: "오늘 오전 근태 기록만" vs "전체")
3. **부분 복구 우선**: 특정 행만 잘못됐다면 전체 복원 대신, 백업에서 해당 데이터만 뽑아 고치는 방식을 개발자와 상의합니다. 전체 복원은 최후의 수단입니다.
4. **별도 환경에서 먼저 검증**: 백업을 **새 임시 Supabase 프로젝트(또는 로컬)** 로 먼저 복원해, 원하는 데이터가 맞는지 확인한 뒤 운영에 반영합니다.
   - 로컬 검증 참고: `supabase db dump`로 스키마/데이터를 받아 로컬(`supabase start`)에서 확인할 수 있습니다. (개발자 작업)
5. **운영 반영은 마지막**: 검증이 끝난 뒤에만 운영 복원을 진행하고, 진행 전 **현재 상태의 즉시 백업**을 한 번 더 만듭니다.
6. **위험 작업 확인**: 운영 DB 복원·삭제·권한 변경은 실행 전 반드시 의도를 재확인합니다(이 저장소 규칙과 동일).

Supabase에서의 복원 위치: **Database → Backups → 원하는 백업의 Restore**. PITR은 **특정 시각 지정 후 Restore**.

## 3. Railway 배포 롤백 방법

증상이 "최근 배포 이후 사이트가 이상함"이라면, 코드/DB를 건드리기 전에 **직전 정상 배포로 되돌리는 것**이 가장 빠르고 안전합니다.

1. https://railway.app 로그인 → 이 프로젝트 → 해당 서비스 클릭.
2. **Deployments** 탭에서 배포 이력을 봅니다. 각 배포에 커밋 메시지가 붙어 있습니다.
3. **직전 정상 배포**를 찾아 오른쪽 **⋯ 메뉴 → Redeploy(또는 Rollback)** 를 누릅니다. → 그 시점 코드로 즉시 되돌아갑니다.
4. 롤백은 **코드만** 되돌립니다. 그 사이 **DB 마이그레이션이 있었다면** 코드만 되돌려선 안 맞을 수 있으니, 마이그레이션이 포함된 배포였는지 개발자와 확인합니다.

> 참고: 배포는 GitHub `master`에 커밋이 올라가면 Railway가 자동으로 빌드합니다(`railway.toml`). 문제가 있는 커밋을 되돌리려면 롤백 후, git에서 해당 변경을 정리하는 것을 개발자에게 요청하세요.

## 4. 장애 시 확인할 로그 위치

| 증상 | 어디서 보나 | 무엇을 보나 |
|---|---|---|
| 사이트가 안 뜸 / 500 오류 | **Railway → 서비스 → Deployments/Logs** | 앱 시작 실패, `console.error` 메시지, 크래시 스택 |
| 로그인/DB 관련 오류 | **Supabase → Logs**(Postgres / Auth / Edge Functions) | 인증 실패, 쿼리 오류, RLS 거부 |
| 인플루언서 수집/분석 오류 | **Supabase → Edge Functions → Logs** | `influencer-extract`/`influencer-analyze`/`push-dispatch` 실행 로그 |
| 헬스체크 실패로 재시작 반복 | **Railway → Deployments** | `/api/health` 응답, restart 이력(`railway.toml`: 최대 10회 재시도) |
| 첫 요청이 느림(콜드스타트) | 정상 동작일 수 있음 | `instrumentation.ts` keepalive 관련, 성능 기준선(`docs/performance/production-baseline.md`) |

> 현재 이 앱에는 Sentry 같은 **오류 자동 알림이 없습니다.** 오류를 능동적으로 알려면 Railway 로그 알림 또는 Sentry(무료 티어) 도입을 개발자와 검토하세요. (별도 개선 항목)

---

## 5. 빠른 점검 체크리스트

- [ ] Supabase **Database → Backups**: 자동 백업 있음? 보존 며칠? PITR 켜짐? → 메모
- [ ] Supabase **Settings → Billing**: 현재 플랜 확인(백업 정책과 직결)
- [ ] Railway **Deployments**: 직전 정상 배포가 무엇인지 눈으로 확인(롤백 대비)
- [ ] 문제가 생기면 → **먼저 로그 확인 → 범위 파악 → 별도 환경 검증 → 마지막에 운영 반영**
- [ ] 운영 DB 복원·삭제·권한 변경은 **실행 전 반드시 재확인**, 진행 전 **즉시 백업 1회 추가**
